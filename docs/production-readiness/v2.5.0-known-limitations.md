# Nightshade v2.5.0 — Known Limitations

_Last updated: 2026-05-11._

This document enumerates items that the v2.5.0 pre-release audit
(`docs/plans/2026-05-09-v250-audit-fixes.md`) identified as worth doing
but explicitly deferred from the v2.5.0 release. Each is tracked for v2.6.

The release is shipping with these limitations because they are either
(a) substantial implementation efforts that would have pushed the release
beyond its hardening window without meaningfully reducing user-visible
risk, or (b) require design discussion that should not be rushed.

## Deferred to v2.6

1. **Web-dashboard feature gap vs the headless API.** The web dashboard
   exposes ~20 endpoints; the headless API exposes 300+. First-class
   panels are still missing for camera cooling, filter wheel, focuser /
   autofocus trigger, rotator, mount slew-to-RA/Dec-by-name, plate-solve
   trigger, sequence load and editor, target catalogue browse, polar
   alignment, flat wizard, mosaic planner, weather / safety monitor,
   dome control, checkpoint resume, profile switching, and session
   analytics. The d-pad, capture, sequence start/stop, and connection
   panels that the dashboard does expose are fully functional. Mobile
   and headless clients have first-class access to all 300+ endpoints
   via the validated API surface added in v2.5.0. Audit reference:
   §2.17.

2. **iOS long-form `UIBackgroundModes`.** v2.5.0 ships the honest path:
   when a sequence starts on iOS, a persistent banner reads "iOS may
   pause monitoring while this app is in the background. Keep the app
   foreground or rely on push from desktop." A real
   `UIBackgroundModes: [audio]` (silent audio keep-alive) or
   `processing` + `BGProcessingTask` solution requires App Store review
   justification and a design discussion that would not have closed
   inside the v2.5.0 hardening window. Push from the desktop side
   continues to work for `image_ready`, `sequence_complete`, and
   `sequence_failed` notifications. Audit reference: §3.2.

3. **Phone-tailored mobile dashboard.** Once connected, the mobile app
   currently uses the shared `NightshadeApp(isMobile: true)` shell —
   i.e., the desktop UI shrunk to phone size. A mobile-first dashboard
   with bottom-tab navigation (Mount / Camera / Sequencer / Live / Log),
   reused business-logic providers from `nightshade_core`, and new
   compact widgets in `apps/mobile/lib/screens/` is queued for v2.6.
   The v2.5.0 mobile experience is functional but cramped on phone-sized
   screens. Audit reference: §3.5.

## Wave 6 (Batch 3) differentiator features — known limitations

The four Batch 3 differentiator features (dynamic scheduler, defect-map
cosmetic correction, plate-solver detection / setup UX, NINA / SGP
sequence import) all ship feature-complete on their end-user surface in
v2.5.0. The items below were flagged by the W6 finisher agents as
worth doing but deferred to v2.5.x, and they are listed here so the
release notes match reality.

1. **Dynamic scheduler — hysteresis weight tuning.** The hysteresis
   ratio compares the final `totalScore` of both candidates
   (`packages/nightshade_core/lib/src/services/scheduler/scheduler_engine.dart:348`).
   That total folds in every scoring weight, including
   `userPriority`. The finisher agent flagged that the default
   `userPriority` weight may need tuning so user priority alone can
   flip the hysteresis without help from other factors (e.g. a target
   the user explicitly bumped to priority 10 should win over a
   marginally higher altitude / lower moon-illumination candidate).
   The default weights ship as a workable starting point; tuning lands
   in v2.5.x. Audit reference: W6-SCHED-FINISH follow-up item 1.

2. **Defect map — `PlatformInt64`/`int` web compatibility.** The
   `lastRebuiltUnixSeconds` field returned from the Rust bridge is a
   `PlatformInt64`. `DefectMapService._fromBridge` assigns it directly
   into the Dart `int` field of `DefectMapStatus`
   (`packages/nightshade_core/lib/src/services/calibration/defect_map_service.dart:85`).
   This is safe on every platform v2.5.0 targets (desktop and mobile
   Dart `int` is 64-bit). It would not compile under web (Dart-on-JS
   `int` is double-precision number and `PlatformInt64` is `BigInt`).
   Web is not a supported target in v2.5.0, so this does not affect
   the release; tracked as W6-DEFECT-FINISH item 2 follow-up so it is
   closed before any future web build.

3. **Plate-solver UX — Verify runs `--help`, not a real solve.** The
   Verify button in Settings → Plate Solving runs
   `<solver> --help` and captures the first non-empty stdout line as a
   version banner
   (`native/nightshade_native/imaging/src/platesolve.rs:294-313`).
   This catches the most common breakage modes (missing binary, wrong
   architecture, missing shared-library dependencies). It is not a
   real end-to-end solve. The orchestrator originally sketched a
   synthetic 200×200 three-cluster FITS fixture for verify, but that
   approach is not viable: ASTAP solves by matching real star fields
   against its installed catalog, so a synthetic fixture cannot solve
   regardless of binary health. A real end-to-end verify path (likely
   bundling a small canonical FITS sample for the verify pass) is
   tracked for v2.5.x. Audit reference: W6-SOLVER-UX-FINISH note.

4. **NINA sequence import — fixtures not from a real NINA export.**
   The NINA test fixtures
   (`packages/nightshade_core/test/services/import/fixtures/nina_basic.json`
   and `nina_unsupported.json`) were authored from the NINA public
   Advanced-Sequencer documentation rather than a verbatim Advanced
   Sequencer JSON export from a current NINA release. The parser,
   canonical-node mapper, summary dialog, persistence, and force-
   import override are all exercised by the fixtures and pass. The
   finisher agent recommends a verification pass against a real
   Advanced Sequencer export before any marketing claim of "imports
   NINA sequences losslessly" — there may be schema fields the public
   docs omit. Audit reference: W6-NINA-IMPORT-FINISH follow-up 1.

## Why these were left for v2.6

The v2.5.0 release was scoped around closing pre-release ship-blocker
risks identified by the 7-agent audit: silent data corruption in the
plate solver and FITS pipeline, the OTA updater's locked-file and
backup behaviour, headless API failures that returned `200 OK` with
`{"status":"failed"}` bodies, and security gaps in the dashboard auth
path, mobile QR pairing, and mobile auth-token storage. Those items all
landed in this release.

The three items above are real limitations — they are not bugs and they
are not silent. The dashboard surfaces what it cannot do; the iOS banner
tells the user the background limit honestly; the mobile UI is usable.
Each item is on the v2.6 roadmap and is the right scope for a feature
release rather than a hardening release.
