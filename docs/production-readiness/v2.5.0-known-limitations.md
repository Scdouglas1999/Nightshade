# Nightshade v2.5.0 — Known Limitations

_Last updated: 2026-05-11._

This document enumerates items that the v2.5.0 pre-release audit
(`docs/plans/2026-05-09-v250-audit-fixes.md`) identified as worth doing
but explicitly deferred from the v2.5.0 release. Each is tracked for v2.6.

The release is shipping with these limitations because they are either
(a) substantial implementation efforts that would have pushed the release
beyond its hardening window without meaningfully reducing user-visible
risk, or (b) require design discussion that should not be rushed.

## Deferred to v2.6

1. **Web-dashboard feature gap vs the headless API.** The web dashboard
   exposes ~20 endpoints; the headless API exposes 300+. First-class
   panels are still missing for camera cooling, filter wheel, focuser /
   autofocus trigger, rotator, mount slew-to-RA/Dec-by-name, plate-solve
   trigger, sequence load and editor, target catalogue browse, polar
   alignment, flat wizard, mosaic planner, weather / safety monitor,
   dome control, checkpoint resume, profile switching, and session
   analytics. The d-pad, capture, sequence start/stop, and connection
   panels that the dashboard does expose are fully functional. Mobile
   and headless clients have first-class access to all 300+ endpoints
   via the validated API surface added in v2.5.0. Audit reference:
   §2.17.

2. **iOS long-form `UIBackgroundModes`.** v2.5.0 ships the honest path:
   when a sequence starts on iOS, a persistent banner reads "iOS may
   pause monitoring while this app is in the background. Keep the app
   foreground or rely on push from desktop." A real
   `UIBackgroundModes: [audio]` (silent audio keep-alive) or
   `processing` + `BGProcessingTask` solution requires App Store review
   justification and a design discussion that would not have closed
   inside the v2.5.0 hardening window. Push from the desktop side
   continues to work for `image_ready`, `sequence_complete`, and
   `sequence_failed` notifications. Audit reference: §3.2.

3. **Phone-tailored mobile dashboard.** Once connected, the mobile app
   currently uses the shared `NightshadeApp(isMobile: true)` shell —
   i.e., the desktop UI shrunk to phone size. A mobile-first dashboard
   with bottom-tab navigation (Mount / Camera / Sequencer / Live / Log),
   reused business-logic providers from `nightshade_core`, and new
   compact widgets in `apps/mobile/lib/screens/` is queued for v2.6.
   The v2.5.0 mobile experience is functional but cramped on phone-sized
   screens. Audit reference: §3.5.

## Why these were left for v2.6

The v2.5.0 release was scoped around closing pre-release ship-blocker
risks identified by the 7-agent audit: silent data corruption in the
plate solver and FITS pipeline, the OTA updater's locked-file and
backup behaviour, headless API failures that returned `200 OK` with
`{"status":"failed"}` bodies, and security gaps in the dashboard auth
path, mobile QR pairing, and mobile auth-token storage. Those items all
landed in this release.

The three items above are real limitations — they are not bugs and they
are not silent. The dashboard surfaces what it cannot do; the iOS banner
tells the user the background limit honestly; the mobile UI is usable.
Each item is on the v2.6 roadmap and is the right scope for a feature
release rather than a hardening release.
