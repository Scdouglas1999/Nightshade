# Nightshade v2.5.0 Pre-Release Audit — Implementation Guide

**Date:** 2026-05-09
**Source:** 7-agent parallel audit (Opus 4.7) of v2.5.0 release branch.
**Status:** All 7 audit reports incorporated. Revised once after a peer review of the headlines.
**Revision:** 2026-05-09 r2 — tightened the plate-solver and FWHM/HFR diagnoses, added a Threat Model section, and replaced the fantasy day-by-day schedule with a phase-by-phase plan and per-area effort estimates.

## How to use this document

Each entry is a self-contained ticket: precise file:line, the problem, the concrete change, and how to verify without hardware. Work top-down inside each section: items are ordered by severity, then by blast radius. Cross-references are inlined.

**CLAUDE.md ground rules govern every fix here:**
- No stubs, no placeholders, no "future update" copy.
- Errors propagate. Silent fallbacks are bugs.
- No `unwrap_or_default`, `let _ =`, `if let Ok` patterns that drop errors. If a fallback is genuinely correct, write a comment that explains *why*.
- Do not write WHAT-style comments; only WHY-style for non-obvious constraints.

## Severity legend

| Sev | Meaning |
|---|---|
| **CRITICAL** | Data loss, hardware risk, user-trapping bug, or release-blocker policy violation. Must ship-block. |
| **HIGH** | Visible quality regression, broken first-run flow, security gap *under the threat model below*, or CLAUDE.md ground-rule violation that affects user-visible behaviour. Strongly recommend fix before release. |
| **MED** | Real but not show-stopping. Schedule for v2.5.1 if 2.5.0 is time-boxed. |
| **LOW** | Polish, dedup, perf nits. Group into a single sweep PR. |

## Threat model (security severities are graded against this)

Nightshade in default mode is a desktop / observatory-LAN application, not a public-internet service. Severity for security items is graded against this assumed threat model:

- **Trusted operator at the controls.** A user holding the desktop session — and any `admin`-scoped bearer token they have minted — is treated as fully trusted. They can already mount-slew, park, abort, and read/write files on the host through the desktop UI; the API is not a privilege boundary against them.
- **Trusted LAN by default.** Headless server defaults to `bindLocalOnly=true`. When the operator opts into `--bind 0.0.0.0`, they are explicitly accepting LAN-attached devices into the trust boundary. The server is not designed to be exposed to the public internet.
- **Untrusted LAN is opt-in.** Pairing flow + bearer-token auth are the protections when a user explicitly chooses to allow non-trusted clients onto the LAN (guest network, AirBnB, conference Wi-Fi).
- **Update package supply chain is hostile by default.** OTA updates and LAN-pushed packages must be cryptographically verified before any byte is written outside a sandboxed staging directory. This is the only piece of Nightshade with a true "fully untrusted source" boundary, and it must be hardened to that bar.

Items the audit ranked HIGH that are downgraded to MED under this threat model (still worth fixing, but not 2.5.0 ship blockers):

- §2.22 Constant-time bearer-token comparison.
- §2.24 `/api/files/browse` admin-scoped path enumeration.
- §2.27 CORS reflection on high-risk POSTs.
- §2.28 WebSocket query-param auth (token in access logs).

Items that remain HIGH under this threat model because they break the default configuration regardless of trust assumptions:

- §2.1 Web dashboard pairing 404 in headless mode (first-run UX is dead).
- §2.7 d-pad runaway slew (default UX is dangerous on any host).
- §2.21 130 unsafe payload casts → HTTP 500 with stack-trace leakage (affects every API client even on a trusted LAN; CLAUDE.md ground-rule violation).
- §2.23 HTTP 200 with `{"status":"failed"}` (breaks every well-behaved API client regardless of trust).
- §3.1 Mobile QR pairing accepts arbitrary JSON (the *whole point* of QR pairing is the trust-establishment step).
- §3.3 Mobile auth token in plaintext SharedPreferences (loss of one phone = full mount/camera control).
- §7A.1–§7A.4 OTA updater (the only true untrusted-source boundary).
- §7A.7 LAN push package size cap (pre-signature filesystem-fill).

## Release-blocker fast pass (do these first)

These are the items that make v2.5.0 unsafe to ship. Many are direct CLAUDE.md violations.

### CRITICAL — silent data corruption / hardware risk / deceptive features

1. **Internal plate solver does not solve.** Three failures stacked: (a) `solve_internal` returns the FITS header / user-supplied hint coordinates verbatim as `ra`/`dec` with no astrometric matching against any catalog; (b) pixel scale is also a header passthrough — never measured; (c) rotation is computed by PCA on detected-star *spatial layout*, which returns the long axis of the star distribution, not the WCS field rotation. The function still returns `success: true`. Combined with `is_solver_available()` hardcoded `true`, every science product downstream is anchored to commanded pointing, not real sky — §6.1, §6.2.
2. **FITS BSCALE/BZERO round-trip is broken.** Read-applies + write-re-emits → double-scaled output. COMMENT/HISTORY cards are also malformed — §6.3, §6.4, §6.5.
3. **OTA updater silently skips locked files** with "size match" optimization, with **partial backup** that can't restore everything the apply step touched, and `cleanup_staging` deletes the **parent** of staging (wiping the backup directory) — §7A.1, §7A.2, §7A.3, §7A.4.
4. **`device_id.rs` `valid_vendors` allow-list is out of sync** with what driver discovery emits. Saved profiles for LX200/Player One/Fujifilm/QHY-CFW/FLI-Focuser/EAF/EFW/Touptek 4-part IDs are rejected on parse — §5.1, §5.2.
5. **Alpaca image-array download silently substitutes `0`** for any pixel JSON value that fails to parse — corrupts entire frames — §5.3.
6. **Sequencer trigger-state writes use `try_write` and drop on contention.** Lost updates: target name/RA/Dec, HFR baseline, exposure count, plate-solve drift, autofocus invalidation — §1.1.
7. **HFR "median" is a noise-biased EMA**; triggers fire on single bad frames — §1.2.
8. **Session checkpoint loses every trigger-state field**; resume can cause **double meridian flip** — §1.3.

### HIGH — visible bugs, security gaps, CLAUDE.md "errors are a feature" violations

9. **`mount_get_status` fabricates defaults** for `at_home`/`sidereal_time`/`pier_side`/`tracking_rate`/`alt`/`az` on read failure — user can't tell "off" from "broken" — §5.4.
10. **SkyWatcher mount returns Ok-with-fake-data** for `set_tracking_rate(Sidereal)`, pier-side, alt/az, sidereal time — meridian flip on SkyWatcher is silently broken — §5.5.
11. **ZWO/QHY/Player One cameras hardcode `cooler_on: false`** in status — dashboard lies after `set_cooler(true)` — §5.7.
12. **FWHM/HFR is over-stated by ~25%.** The code applies `FWHM = 2.3548 × HFR`, but the underlying `calculate_hfr_at_point` computes flux-weighted *mean* radius ⟨r⟩ (`weighted_radius_sum / total_flux`), not the true encircled-energy-50% half-flux radius. For a Gaussian: ⟨r⟩ ≈ 1.253σ, true HFR ≈ 1.177σ, FWHM ≈ 2.355σ → reported FWHM is ~25% too large. Fix is *pick one definition and use the matching factor*; do not just change the constant — §6.11.
13. **VNG debayer threshold is degenerate** (always passes all 8 directions → no edge preservation) — every OSC frame loses sharpness — §6.12.
14. **Trigger-level `RecoveryAction::Retry` is a no-op**; default "Guiding Failure" trigger silently does nothing — §1.5.
15. **Two parallel meridian-flip implementations diverge** — §1.6.
16. **Two parallel `AutofocusConfig` types** silently drop user-tunable fields like `backlash_compensation` — §1.7.
17. **`UpdateDitherConfig`/`UpdateLocation`/`UpdateFilterOffsets` runtime commands are stubs that only `tracing::info!`** — §1.8.
18. **`FlipFailureAction::AbortAndPark` silently drops the park error** — the worst-case astrophotography scenario goes uncommunicated — §1.10.
19. **Web dashboard "Pair" button 404s in headless mode** — first-run web onboarding is dead in the most common deployment — §2.1.
20. **Web dashboard mount d-pad slews indefinitely on `click`** (no `pointerup` stop) — §2.7.
21. **Mobile QR pairing accepts arbitrary JSON**; **iOS sessions silently die in background** — §3.1, §3.2.
22. **Equipment screen ships a "future update" placeholder dialog** on every connected device card — direct CLAUDE.md violation — §4.1.
23. **Centering dialog has no abort path during run** — §4.2.
24. **Headless API: 130 unsafe `as String` casts** return 500 with `e.toString()` leaking internals — §2.21. (HIGH regardless of threat model: every API client breaks on missing/wrong-type fields with a stack-trace dump.)
25. **Headless API: HTTP 200 with `{"status":"failed"}` body** for real failures across backup/flat-wizard/mosaic/sequence-mgmt handlers — §2.23. (HIGH: breaks every well-behaved REST client and observability tooling.)
26. **Behavioral audit + fail-closed audit cover only a hardcoded ~17-file allowlist**; every new headless handler, the updater, and the LAN push receiver are invisible to the gates — §7B.1, §7B.3. (HIGH: false-confidence release-gate.)

(Items §2.22 constant-time tokens, §2.24 admin-scoped FS browse, §2.27 CORS reflection, §2.28 WS-query auth are graded MED under the trusted-LAN threat model in the section above. Fix them, but they don't ship-block.)

# 1. Sequencer engine (Rust)

Crate: `native/nightshade_native/sequencer/`

## 1.1 [CRITICAL] Replace `try_write` storm on trigger state

**Files:** `node.rs:743, 781, 854, 934, 1023, 1560`; `instructions.rs:746`.

**Problem:** Trigger-state mutations from instruction code use `try_write()`. The trigger monitor task holds a read lock once per second; any contention drops the update silently. Lost updates include target name, target RA/Dec, HFR baseline, exposure count, plate-solve drift baseline, autofocus invalidation, and completed-integration counter.

**Fix:**
1. Grep the sequencer crate for `state.try_write()` and replace with `state.write().await` at every site listed above. Functions are already `async`; tokio's `RwLock::write()` is FIFO-fair and the critical sections are tens of microseconds.
2. Confirm no caller is in a non-async closure (none of the listed sites are).
3. Audit any remaining `try_write`/`try_read` in the sequencer crate. Either justify with a `// Why:` comment (e.g., "best-effort metric publish") or convert to the awaiting form.

**Verify:**
- Add a unit test in `executor.rs` that spawns a fake trigger monitor reading every 10 ms and 32 instruction tasks each writing the target name; assert all writes are observed.
- `cargo clippy -p nightshade_sequencer --all-features -- -D warnings`.

## 1.2 [CRITICAL] Fix HFR "median" computation

**File:** `node.rs:783-786`.

**Problem:** Code labelled "median_hfr" computes `((a+b)/2 + c)/2 + d)/2` — an exponentially-weighted moving average where the latest frame counts ½, the previous ¼, etc. HFR-degraded triggers therefore react to a single noisy frame instead of a true central tendency.

**Fix:**
1. Sort `result.hfr_values` (clone-and-sort or `sort_by` with `partial_cmp` for f64 NaN safety).
2. Take `values[len/2]` for odd length, `(values[len/2 - 1] + values[len/2]) / 2.0` for even length.
3. Filter out `NaN` and any value `<= 0.0` before sorting.
4. Rename the field/local from `median_hfr` to match what's stored if you choose mean instead.

**Verify:**
- Unit test: `[1.5, 1.6, 1.7, 1.8, 5.0]` → median 1.7 (current code returns ~3.2).
- Unit test with NaN/zero filtered.

## 1.3 [CRITICAL] Persist trigger state in session checkpoints

**File:** `checkpoint.rs:16-48` (`SessionCheckpoint`).

**Problem:** Resume after crash loses: `last_autofocus_frame`, `last_dither_frame`, `baseline_hfr`, `has_flipped_this_target`, `tracking_limit_detected_at`, `current_filter`, `grid_dither_index`, `triggers_enabled`, `safety_fail_mode`, `filter_focus_offsets`. After resume, `AutofocusInterval` and `DitherInterval` misfire and a target that already flipped can double-flip.

**Fix:**
1. Define a serde-`Serialize`/`Deserialize` `TriggerStateSnapshot` struct in `checkpoint.rs` mirroring those fields from `triggers.rs::TriggerState`.
2. Add `pub trigger_state: Option<TriggerStateSnapshot>` to `SessionCheckpoint` (Option for backward-compat reads).
3. On save: snapshot the current `TriggerState` via a new `TriggerState::snapshot(&self) -> TriggerStateSnapshot` method.
4. On resume: in `resume_from_checkpoint`, before starting the trigger monitor task, call `TriggerState::restore_from(snapshot)`.
5. Bump `CHECKPOINT_VERSION` and write a one-line migration: missing field → defaults (current behaviour, but logged at `info!`).

**Verify:**
- Round-trip test: build `TriggerState`, serialize → deserialize, assert equality on every field.
- Integration test: pause sequence with `has_flipped_this_target = true`, save checkpoint, restore, assert flag survives.

## 1.4 [HIGH] Sort out `executor.rs:1142-1170` SafetyFailMode dead branches

**File:** `executor.rs:1142-1170`; `sequencer_api.rs:160-167`.

**Problem:** All three `SafetyFailMode` variants (`FailOpen`/`FailClosed`/`WarnOnly`) collapse to the same `false` branch. The set-API also coerces non-FailClosed to FailClosed. The enum is a public-API lie.

**Fix:**
- **Pick one path:**
  - **(A) Implement them.** `FailOpen` → continue on poll failure with one warning event. `WarnOnly` → never abort; emit alarm. `FailClosed` → current behaviour. Document each in `SafetyFailMode` doc comment.
  - **(B) Delete `FailOpen`/`WarnOnly` from the enum and from `sequencer_api.rs`.** Remove the API coercion. Update FRB bindings (`melos run generate`).
- **Recommended: (A).** Real users want WarnOnly during shutdown sequences and dawn testing.

**Verify:**
- Unit test the executor poll path with all three modes and a synthetic poll failure; assert correct branch is taken.

## 1.5 [HIGH] Implement trigger-level `RecoveryAction::Retry`, add `Dither`, decide on `Continue`/`CustomBranch`

**Files:** `executor.rs:1448-1702` (the trigger handler `match &action`); `lib.rs` (`RecoveryAction`); `triggers.rs:1009-1015` (`DitherInterval` standard trigger).

**Problem:** The match ends in `_ => {}`, silently dropping `Retry`, `Continue`, and `CustomBranch`. The default standard "Guiding Failure" trigger uses `Retry { max_attempts: 3 }` — at runtime this does nothing. Separately, `TriggerType::DitherInterval` is defined but never instantiated as a standard trigger and there is no `RecoveryAction::Dither`.

**Fix:**
1. **Add `RecoveryAction::Dither(DitherConfig)` variant.** Wire its handler to call `execute_dither` from the trigger executor with the same context the instruction node uses.
2. **Implement `RecoveryAction::Retry { max_attempts }`** for triggers: on fire, pause the current exposure, run the trigger-specific recovery (e.g., for guiding: stop guiding, restart, wait for settle), retry up to `max_attempts`; on exhaustion, abort with reason.
3. **Decide `Continue` and `CustomBranch`:** if they are node-only, return them as compile error in trigger config validation (instead of `_ => {}`), or implement them — typically `CustomBranch` runs a child node attached to the trigger.
4. **Instantiate the `DitherInterval` standard trigger** in `triggers.rs::default_triggers` (or wherever standard triggers are built) so `RecoveryAction::Dither` actually fires.

**Verify:**
- Unit test that fires a synthetic `GuidingFailed` trigger with `Retry { max_attempts: 2 }` and asserts a stop/start cycle on the (mocked) PHD2 backend.
- Integration test that adds a `DitherInterval` trigger and asserts the dither instruction runs at the configured cadence.

## 1.6 [HIGH] Unify the two meridian-flip implementations

**Files:** `instructions.rs::execute_meridian_flip` (~lines 2847-3300, ~1000 LOC); `meridian_flip_executor.rs::MeridianFlipExecutor` (~870 LOC).

**Problem:** Two complete implementations are reachable: explicit `MeridianFlip` instruction nodes call the first; `RecoveryAction::MeridianFlip` calls the second. They differ in: timeout values, post-flip altitude check, autofocus method (executor uses hardcoded steps_out=7/step_size=100/exposure=3.0), settle behaviour, plate-solve failure handling (instruction = warn, executor = Failed → execute_failure_action), pier-side telemetry fallback (instruction handles unavailable, executor returns Unknown), abort-during-flip (instruction restores tracking, executor leaves tracking off), `mark_flip_performed` (only the trigger path marks it).

**Fix:**
1. Pick `MeridianFlipExecutor` as the canonical engine (it is newer, has `MIN_POST_FLIP_ALTITUDE_DEG`, has structured failure actions).
2. Backport the missing instruction-path behaviours into `MeridianFlipExecutor`:
   - Coordinate-fallback when pier-side telemetry is unavailable (port `verify_pier_side_changed` to fall back to RA/Dec comparison).
   - Tracking-restore on cancel.
   - Pull autofocus parameters from the user equipment profile, not hardcoded constants (`meridian_flip_executor.rs:529-538`).
   - Plate-solve-failure-after-flip-success warning behaviour (do not invoke `execute_failure_action` if the flip itself succeeded; warn instead).
3. Refactor `instructions.rs::execute_meridian_flip` to a thin wrapper that builds a `MeridianFlipExecutor` and calls `execute()`.
4. Always call `state.mark_flip_performed()` on success in the unified path.
5. Delete the local `julian_day` in `meridian_flip_executor.rs`; reuse `meridian::julian_day`.

**Verify:**
- Move existing meridian unit tests to target the unified engine.
- Add tests for: pier-side fallback, cancel-during-settle, autofocus params from profile.

## 1.7 [HIGH] Unify the two `AutofocusConfig` types

**Files:** `lib.rs:553` (user-facing); `autofocus.rs:45` (engine).

**Problem:** User config lacks `backlash_compensation`, `outlier_rejection_sigma`, `use_temperature_prediction`, `max_star_count_change`. `instructions.rs:1097` builds an engine config from defaults and copies only 4 fields; user UI cannot tune the rest. Backlash always uses 50 steps regardless of user input.

**Fix:**
1. Make the engine config the source of truth: add the missing fields to the user-facing struct.
2. In `instructions.rs:1097`, copy *all* fields (consider implementing `From<UserAutofocusConfig> for AutofocusEngineConfig`).
3. Update FRB bindings (`melos run generate`).
4. Surface the new fields in the autofocus UI (`packages/nightshade_app/lib/screens/imaging/tabs/focus_tab.dart` — verify field list).

**Verify:**
- `cargo check -p nightshade_sequencer && cargo check -p nightshade_bridge`.
- Manual: equipment profile JSON round-trip preserves the new fields.

## 1.8 [HIGH] Implement (or remove) `UpdateDitherConfig` / `UpdateLocation` / `UpdateFilterOffsets` runtime commands

**Files:** `executor.rs:1054-1084`, `executor.rs:1813-1846`, `executor.rs:1828` (`let _ = (pixels, settle_pixels, ...)`).

**Problem:** These commands `tracing::info!` and discard. Comment claims "values take effect on next sequence load" — a stub disguised as a feature, forbidden by CLAUDE.md.

**Fix:**
1. Add a shared `Arc<RwLock<RuntimeConfig>>` that the spawned executor task and instruction nodes read from on each invocation.
2. `RuntimeConfig` carries `dither_config: DitherConfig`, `location: Location`, `filter_offsets: HashMap<String, i32>`.
3. `execute_dither`, `execute_capture` (filter selection), and any altitude-aware logic read from this Arc instead of from cloned-at-load values.
4. `UpdateDitherConfig`/`UpdateLocation`/`UpdateFilterOffsets` write through the Arc and emit a `RuntimeConfigUpdated` event.
5. Remove `let _ = (pixels, settle_pixels, ...)` (`executor.rs:1828`).

**Verify:**
- Unit test: build executor, send `UpdateDitherConfig`, run a dither, assert new pixel count is honoured.

## 1.9 [HIGH] Reset `has_flipped_this_target` on observable pier-side change

**File:** `triggers.rs:838-847` (`set_meridian_target`).

**Problem:** Flag is reset only when target name changes. A long single-target session that crosses two meridians (high-latitude / pause-resume / mosaic-with-shared-name) silently skips the second flip.

**Fix:**
1. Add `TriggerState::on_pier_side_observed(side: PierSide)`. When the observed side returns to the pre-flip side after `has_flipped_this_target` was set, clear the flag.
2. Call from the trigger monitor wherever pier-side is read.
3. Add an explicit `clear_flipped_state()` method called by the meridian-flip post-flip path on success and on any natural target-boundary in a sequence.

**Verify:**
- Unit test: simulate flip, then a second pier-side change back, assert flag clears.

## 1.10 [HIGH] Stop dropping park errors after flip failure

**File:** `meridian_flip_executor.rs:683-710` (`execute_failure_action`); also `:381, 489, 499, 687, 698-700` (`let _ = abort_slew(...)`, `mount_set_tracking`, `mount_park`).

**Problem:** When the failure action is `AbortAndPark`, a park failure means the mount may be at a hard limit. Code drops the error with `let _ = ...`. This is the worst-case scenario in astrophotography and the most-egregious silent-fallback in the file.

**Fix:**
1. Replace each `let _ = mount_park(...)` with explicit handling: log at `error!`, retry park up to N times with a short delay, emit a `CriticalSafetyEvent` to the event bus that the UI surfaces as a top-level alert.
2. Same for `mount_set_tracking(false)` and `abort_slew` — log at `error!`, do not silently drop.
3. If retries exhaust, return `Err` so the executor fault state propagates (instead of pretending the failure action succeeded).

**Verify:**
- Unit test with a mocked mount that fails park; assert the executor returns `Err`, an event is emitted, retries happened.

## 1.11 [HIGH] Wire `DriftLimit` plate-solve-drift trigger

**File:** trigger evaluator in `triggers.rs`; drift calc already exists in `instructions.rs::calculate_drift_pixels`.

**Problem:** Drift detection logic is present but no trigger fires on it. Competitors all have this.

**Fix:**
1. Add `TriggerType::DriftLimit { max_pixels: f64 }` variant.
2. In the trigger monitor, when a plate-solve completes, compute drift and fire if exceeded.
3. Default standard trigger entry; default `RecoveryAction::Recenter` (re-slew + plate-solve).

**Verify:** Unit test that synthesises a plate-solve event with drift past threshold and asserts the trigger fires.

## 1.12 [MED] Stop silently dropping unwrapped tree children

**File:** `node.rs:1944-1956`.

**Problem:** `Arc::try_unwrap` failure path explicitly comments "leave it out". Tree corruption silently masked.

**Fix:** Return a `Failed` status (or `panic!` if this branch indicates a logic bug — strongly prefer Failed with an event so the user is told). Replace the "leave it out" comment with a `// Why:` explaining the invariant.

## 1.13 [MED] Fix dither RA-only fallback

**File:** `instructions.rs:1626`.

**Problem:** `if dec_offset.abs() < 0.01` collapses to RA-only dither even when the user requested grid pattern; magic threshold and behaviour mismatch.

**Fix:** Skip dither explicitly when both offsets are below threshold (already handled at line 1613); for grid mode, never coordinate-magic to RA-only — pick the nearest grid cell.

## 1.14 [MED] Streaming-checkpoint task should not live inside trigger monitor

**File:** `executor.rs:1404`.

**Problem:** Streaming checkpoint cadence depends on trigger monitor liveness. If `triggers_enabled = false`, no checkpoints are saved.

**Fix:** Spawn an independent `tokio::time::interval(30s)` task. Owned by the executor, joined on shutdown. Use the same `Arc<CheckpointManager>` (see §1.16).

## 1.15 [MED] Filename derivation must not silently substitute defaults

**Files:** `instructions.rs:594-598` (`unwrap_or("image")`, `unwrap_or_default()` for parent), `:871, 988-998` (filter substitutions: `unwrap_or("unfiltered")`, `unwrap_or("L")`).

**Problem:** Bad input silently produces "unknown" file layouts; calibration captures can be mis-labelled "L".

**Fix:** Treat missing fields as configuration error → return `Err`. If a default is genuinely safe (e.g., "image" for a missing target name during a daytime test capture), log at `warn!` and document `// Why:`. Never silently substitute a filter name.

## 1.16 [MED] One CheckpointManager per session, shared via Arc

**File:** `executor.rs:756-757`.

**Problem:** Streaming task creates a second `CheckpointManager` for the same dir, bypassing the `info_cache`. UI staleness on `has_recoverable_checkpoint`.

**Fix:** Pass `Arc<CheckpointManager>` into the streaming task. Mark `info_cache` as `Arc<RwLock<...>>` if needed.

## 1.17 [MED] `MeridianFlipExecutor::wait_settle` must check executor cancellation

**File:** `meridian_flip_executor.rs:614`.

**Problem:** Settle only checks abort, not the executor's `is_cancelled`. Stop during settle waits to settle completion.

**Fix:** Take an `Arc<AtomicBool>` (or `CancellationToken`) and check inside the settle loop. Return early on cancel.

## 1.18 [MED] Verify `is_cancelled` is set before `return fired_triggers` in all trigger-monitor exit paths

**File:** `executor.rs:1497, 1684` (`RecoveryAction::ParkAndAbort`, `FlipFailureAction::AbortAndPark`).

**Problem:** Currently OK in the inspected paths, but the invariant is implicit. A future `match` arm could regress.

**Fix:**
1. Extract a `terminate_with(triggers, reason)` helper that sets `is_cancelled = true` and returns `triggers`.
2. Replace every `return fired_triggers;` in this function with the helper.
3. Add a doctest example of correct usage.

## 1.19 [MED] Pre-flip cover-state check

**Files:** both meridian-flip implementations.

**Problem:** Neither path verifies the dust cap is open. Post-flip plate solve fails on covered camera, triggering `AbortAndPark` unnecessarily.

**Fix:** Before starting the flip, query the cover/calibrator device state if one is connected. If covered, refuse to flip with a clear error event. Document the case in `MeridianFlipExecutor::execute` rustdoc.

## 1.20 [MED] `meridian_flip_executor.rs:185-187` retry-delay magic 60.0

**Problem:** `.unwrap_or(60.0)` when `retry_delays_secs` array is shorter than the attempt index. User configured 30s gets ignored.

**Fix:** Use `.last().copied().unwrap_or_else(|| return Err(ConfigError))` (return error if user has empty array). Document the saturating behaviour with a `// Why:` comment.

## 1.21 [LOW] Remove redundant tracing of `.unwrap_or_default()`

**File:** `triggers.rs:270-286`.

**Problem:** Tracing prints `0.0` for unknown hour-angle as if it were data.

**Fix:** Change to `Option::map(|v| format!("{:.3}", v)).unwrap_or_else(|| "n/a".into())` in the tracing field.

## 1.22 [LOW] Configurable retention windows + min-star-count

**Files:** `triggers.rs:644-655` (RMS history 5 min hardcoded); `instructions.rs:1178` (`MIN_STAR_COUNT = 10`); `triggers.rs:1108-1121` (`FocusDrift` defaults).

**Fix:** Move to fields on the relevant config struct; default to current values.

## 1.23 [LOW] Comments-as-WHAT cleanup

Sequencer crate is pervasive with WHAT-style comments (e.g., `executor.rs:130`, `:1059-1061`; `meridian_flip_executor.rs:75`). Schedule a sweep PR after the higher-severity items.

## 2.1 [HIGH] Web dashboard "Pair" button 404s in headless mode

**Files:** `apps/desktop/web_dashboard/js/api.js:380-387`; `apps/desktop/lib/headless_api_server.dart:200-217, 913`.

**Problem:** Dashboard calls `POST /api/pairing/verify`. `HeadlessApiServer` reports `pairingSupported: false` and has no such route. First-run web onboarding is dead in the most common deployment mode — users must read the bearer token from console output.

**Fix (preferred):** implement pairing in headless server.
1. Move `TokenManager` and `PairingDatabase` (currently in `nightshade_webrtc/lib/src/web_server.dart`) into `apps/desktop/lib/headless_api/auth/` (or `packages/nightshade_core` if both servers will use it).
2. Add `POST /api/pairing/start` (returns 6-digit code shown on desktop console) and `POST /api/pairing/verify` to `HeadlessApiServer`.
3. On verify, mint a long-lived bearer token, persist with the device fingerprint, return JSON `{ token, expiresAt }`.
4. Set `pairingSupported: true` in `/api/info`.
5. Lock pairing attempts: 5 per 10-minute window (already a pattern in `web_server.dart:301-302` — port the LRU eviction along with it).

**Fix (fallback):** hide the UI when unsupported.
- In `app.js`, gate the entire Pairing tab on `info.pairingSupported === true`. Show a clearer "paste your token from terminal" affordance otherwise.

**Verify:** Manual flow from a fresh headless launch: open dashboard → see Pair UI → enter code → token persists; refresh, no re-pair needed.

## 2.4 [HIGH] Restrict dashboard CSP `connect-src`

**File:** `apps/desktop/web_dashboard/index.html:5`.

**Problem:** CSP allows `connect-src 'self' http://*:* https://*:* ws://*:* wss://*:*`. Dashboard served from server A can be tricked to talk to attacker-controlled server B.

**Fix:**

## 2.21 [HIGH] Replace 130 unsafe payload casts with a `validatePayload` helper

**Files:** every file under `apps/desktop/lib/headless_api/handlers/`.

**Problem:** `payload['x'] as String` / `(x as num).toDouble()` patterns return HTTP 500 with `e.toString()` (leaking stack traces and internal types) on missing or wrong-type fields.

## 2.22 [HIGH] Constant-time bearer-token comparison

**File:** `apps/desktop/lib/headless_api_server.dart:2607-2612` (`_scopeForToken`).

**Problem:** Plain `Map[]` lookup. Timing-attack observable; allows an attacker to extract tokens character-by-character on a slow link.

**Fix:**
1. Iterate the full `_effectiveAuthTokensByValue` map; compare each token with the request's bearer using `_constantTimeCompare` (already implemented in `lan_push_receiver.dart`).
2. Move `_constantTimeCompare` into a shared `apps/desktop/lib/headless_api/auth/timing.dart` and reuse.
3. Token-bucket rate-limit token-comparison failures so the loop cost can't be turned into DoS.

**Verify:** Microbenchmark: time-difference between a valid token and a 1-char-different token must be < the token-comparison loop's natural variance.

## 2.23 [HIGH] Stop returning HTTP 200 with `{"status":"failed"}`

# 6. Imaging pipeline + science analytics

## 6.1 [CRITICAL] Internal plate solver does not solve

**File:** `native/nightshade_native/imaging/src/platesolve.rs:952-978` (`solve_internal`); rotation estimator at PCA-on-centroids.

**Problem:** Three layered failures, each independently wrong:
1. **Position is a passthrough.** `infer_center_from_header(&header, hint_ra, hint_dec)` returns the FITS header `RA`/`DEC` if present, else the user-supplied hint. No image data is consulted for the center. Returned `ra`/`dec` is whatever the mount or sequencer asked for.
2. **Pixel scale is also a passthrough.** `infer_pixel_scale_from_header` reads `FOCALLEN`/pixel-size from headers — never measured against the actual frame.
3. **Rotation is the wrong algorithm.** `estimate_rotation` runs PCA on detected-star centroids, which returns the long axis of the star spatial distribution — not the WCS field rotation. A field full of randomly-distributed stars will produce different "rotation" from frame to frame regardless of actual orientation.

Stars are detected (line 964) but only used to feed the broken rotation calc. The function still returns `success: true`. All downstream science (photometric calibration, transparency, residuals, moving-object detection, line-ratio products) treats this WCS as truth. Direct CLAUDE.md violation: "errors are a feature."

**Fix (recommended for v2.5.0):**
1. Delete `solve_internal` from the public solve path.
2. Make `solve_with_astap` and `solve_with_astrometry` the only paths.
3. Update `is_solver_available()` (per §6.2) to actually check.
4. UI: when no external solver is available, show a guided install dialog ("Plate solving requires ASTAP. Download from <link>; configure path in Settings.").

**Fix (long-term, v2.6+):**
1. Implement a real geometric matcher: 4-star quad hashing (Astrometry.net algorithm) or triangle matching against a Tycho-2 / Gaia-DR3 subset shipped with the app (~50-200 MB depending on magnitude limit).
2. Compute true CD matrix from at least 4 matched pairs (least-squares fit).
3. Validate solve quality with residuals; reject if RMS > N pixels.

**Verify:** Run plate solve against a real ASTAP-solved frame's pixel data with the previous WCS scrubbed; assert RA/Dec within 5 arcsec of the known solution.

## 6.2 [CRITICAL] `is_solver_available()` must reflect reality

**Problem:** Returns hardcoded `true`. Combined with §6.1, this advertises a working solver that doesn't exist.

## 6.3 [CRITICAL] FITS BSCALE/BZERO round-trip is broken

**File:** `native/nightshade_native/imaging/src/fits.rs:541-564, 199-272`.

**Problem:** Read applies BSCALE/BZERO to data buffer (lines 199-272) and stores the original BSCALE/BZERO in `header.keywords`. On write (line 541-564), a new `BZERO=32768/BSCALE=1` is forced for U16, but the original BSCALE/BZERO entries in `keyword_order` are not stripped — round-trip of a `BSCALE=2.0/BZERO=1000.0` file produces a doubly-scaled FITS. For non-U16 paths, similar issue.

**Fix:**
1. After applying BSCALE/BZERO during decode, **strip both keywords** from `header.keywords` and `keyword_order`. The in-memory data is now "unscaled" relative to the stored values; the rewritten file must reflect that.
2. On write: emit only the freshly-computed `BSCALE`/`BZERO` for the chosen output BITPIX. Never write through stale source values.
3. Add a round-trip test: load an ASTAP-solved real-world FITS with non-trivial BSCALE/BZERO; write to a temp file; reload; assert pixel-array equality and header self-consistency.

## 6.4 [CRITICAL] Silent `unwrap_or(0.0)` in WCS parsing

**File:** `native/nightshade_native/imaging/src/platesolve.rs:309-313, 396-401`.

**Problem:** Every CRVAL1/CRVAL2/CD1_1/CD1_2/CD2_1/CD2_2 parse uses `.unwrap_or(0.0)`. A malformed WCS line yields a "successful" solve at RA=0/Dec=0/scale=0.

## 6.5 [HIGH] FITS header — string padding, comment cards, COMMENT/HISTORY storage

**File:** `native/nightshade_native/imaging/src/fits.rs:170-197, 330-335, 541-564`.

**Problem (multiple):**
1. String values are emitted as `'value'` without FITS-mandated padding to 8 chars between the quotes (FITS 4.2.1.1). PixInsight, AstroPixelProcessor, and some legacy tools reject sub-8-char strings.
2. `FitsValue::Comment(c)` is written via `write_keyword(key, c)` producing `KEYNAME = comment_text` — malformed COMMENT card.
3. COMMENT/HISTORY are stored under synthetic keys `COMMENT_<n>` (>8 chars) and re-emitted on writeback as illegal value cards.
4. No comment field support on regular keywords — the standard allows `KEYNAME = value / comment text`.
5. NAXIS3 silently accepted as 1 if missing; NAXIS=4 (4D cube) loads only the first plane silently.

**Fix:**
1. Pad string values to ≥8 chars between single quotes per FITS spec.
2. Special-case `FitsValue::Comment(c)` and `FitsValue::History(c)`: emit `COMMENT ` + text padded to col 80; no `=` separator. Same for `HISTORY`.
3. Refactor header storage: separate `Vec<String> comments` and `Vec<String> history` from `keywords: BTreeMap<String, FitsValue>`. Drop the `COMMENT_<n>` synthetic keys.
4. Add a `comment: Option<String>` field to FITS keyword writes. Truncate or wrap if total card > 80 bytes.
5. Reject `NAXIS > 3` explicitly with `FitsError::Unsupported4DCube`. If 4D support is desired, implement it; otherwise be honest.

**Verify:**
1. Round-trip test against a real-world `*.fit` from ASTAP / NINA — read, write, load in PixInsight (manual step) and confirm no warnings.
2. Test with NAXIS=4 input — assert `Err`.

## 6.9 [HIGH] Auto-stretch must be MAD-based, matching PixInsight STF

**File:** `native/nightshade_native/imaging/src/stretch.rs:69-92, 37-106`.

**Problem:** Hard-coded fixed-percentile clip (0.001/0.999). On faint nebula or LRGB integration with low background, percentile-clip pegs to bright stars and produces a flat appearance. The MAD/median structure is computed but discarded — only `median` is used.

**Fix:**
1. Compute `MAD = median(|x - median|)`.
2. Shadow clip: `c0 = clip(median - 2.8 * 1.4826 * MAD, 0.0, 1.0)` (the `1.4826` is `1 / Φ⁻¹(0.75)`, the standard MAD-to-σ scale).
3. Highlight clip: `c1 = 1.0` (or `median + 8 × 1.4826 × MAD`, capped at 1.0).
4. Mid-tone target: `m = -2.8 + (1.0 - target_brightness)` (typical target_brightness = 0.25).
5. Apply per channel for color, or use the brightest-channel parameters across all channels for "linked" stretch.
6. Match PixInsight's STF reference; document the formula in a `// Why:` block.

**Verify:** Visual diff against PixInsight STF on the same input frame; should look indistinguishable for typical narrowband and LRGB inputs.

## 6.10 [HIGH] SNR formula must use the standard CCD equation

**File:** `native/nightshade_native/imaging/src/stats.rs:481-486`.

**Problem:** `sum_flux / (noise * sqrt(sum_flux / peak_above_bg))` — non-standard. Omits read noise and dark current. Uses pixel-count-via-flux-ratio as a proxy for aperture pixel count. Reported SNR is systematically too optimistic.

**Fix:** `SNR = signal / sqrt(signal + n_pix * (sky + read_noise² + dark))`.
1. Pass `n_pix` (actual aperture pixel count from the detection mask) into the function.
2. Require `read_noise_e` and `dark_current_e_per_sec * exposure_s` as inputs (from camera metadata or quirks DB).
3. Convert ADU to electrons via `gain_e_per_adu` first.
4. Document the model in a doc-comment.

## 6.11 [HIGH] Fix FWHM / HFR formulas

**Files:** `native/nightshade_native/imaging/src/stats.rs:464-543, 477`.

**Problem:** Two compounding errors. Either alone would over-state seeing; together they make every FWHM ~25% too large, which is then written to `science_frame_quality_metrics` and PSF map tiles.

1. `calculate_hfr_at_point` (`stats.rs:512-543`) computes `weighted_radius_sum / total_flux` — the **flux-weighted mean radius** ⟨r⟩, not the half-flux (encircled-energy 50%) radius. Convention in NINA / SGP / PixInsight is the genuine 50% encircled-energy radius. The two are close for Gaussians but diverge for Moffat / diffraction-limited profiles.
2. `stats.rs:474-478` then applies `FWHM = 2.3548 × HFR` (the FWHM/σ ratio), assuming the value at hand is σ. It is not σ — it is ⟨r⟩.

For a Gaussian:
- ⟨r⟩ = σ√(π/2) ≈ 1.253σ
- True HFR (50% encircled-energy) = σ√(2 ln 2) ≈ 1.177σ
- FWHM = 2σ√(2 ln 2) ≈ 2.355σ
- ⇒ correct conversion if you keep ⟨r⟩: FWHM ≈ 1.879 × ⟨r⟩
- ⇒ correct conversion if you switch to true HFR: FWHM ≈ 2.000 × HFR_true

The current code's `FWHM = 2.355 × ⟨r⟩` ≈ 2.951σ, ~25% too large.

**Fix — pick one and use the matching factor:**

**Option A (recommended):** Switch to true encircled-energy 50% HFR.
1. In `calculate_hfr_at_point`, sort pixels in the detection aperture by distance from centroid; integrate background-subtracted flux in increasing-radius order; return the radius at which cumulative flux ≥ 50% of total. Linearly interpolate between the bracketing pixels for sub-pixel accuracy.
2. Apply `FWHM = 2.0 × HFR`.
3. Match NINA / SGP / PixInsight conventions; cross-comparison becomes meaningful.

**Option B (cheaper):** Keep flux-weighted mean radius but document and convert correctly.
1. Rename the field/metric to `mean_radius` (or document inline that "HFR" here means flux-weighted mean radius).
2. Apply `FWHM = 1.879 × mean_radius` for a Gaussian PSF; document the assumption.
3. Note the reported metric will not match NINA/SGP HFR values directly.

**Verify (load-bearing):** Add a unit test that builds a synthetic Gaussian PSF at known σ on a 64×64 grid, runs the detector, and asserts:
- The reported HFR matches the chosen definition's expected value to within 5% (1.177σ for true HFR; 1.253σ for flux-weighted mean).
- The reported FWHM matches `2.355σ` to within 5%.
A second test with a Moffat-2 profile (β = 2) confirms the metric behaves reasonably under non-Gaussian PSFs.

## 6.12 [HIGH] VNG debayer threshold is degenerate

# 7. OTA updater + production audit tooling

The headless API portion of §7 was covered in §2. This section is the OTA updater + the `tools/production/` audit programs.

## 7A. OTA Updater

Files: `native/nightshade_native/updater/src/main.rs`, `packages/nightshade_updater/lib/`, `apps/desktop/lib/widgets/update_manager.dart`.

### 7A.1 [CRITICAL] Updater must not silently skip locked files

**File:** `native/nightshade_native/updater/src/main.rs:243-249`.

**Problem:** Same-size files get a "MoveFileEx" optimistic replace; on failure (file locked / sharing violation), they're recorded to `skipped_files` and the apply continues. **Different content, same size = silently corrupted install** with no error → no rollback.

**Fix:**
1. Drop the size-match-skip optimization. Always replace.
2. On Windows: if `MoveFileEx` returns `ERROR_SHARING_VIOLATION`, retry up to 3 times with 500 ms delay; if still failing, use `MOVEFILE_DELAY_UNTIL_REBOOT` and write a marker file the next-boot logic consumes — *and* return non-zero from the apply step so rollback runs immediately or the user is told an OS reboot is required.
3. Skipped files become a fatal error, not a warning. Rollback fires. The user is shown an actionable message: "Update could not replace foo.dll because it is in use. Please close Nightshade and try again."

**Verify:** Synthetic test: lock a file in the install dir, run the updater, assert non-zero exit, assert backup is restored.

### 7A.2 [CRITICAL] Updater must back up every file the apply step touches

**File:** `native/nightshade_native/updater/src/main.rs:177-208, 211-271`.

**Problem:** Backup covers only 3 specific EXEs/DLLs and the `data/` folder. `apply_update` walks the entire staging tree and writes anywhere under `install_dir`. Files added or modified outside the backup set cannot be reverted.

**Fix:** **Pick one strategy:**
1. **Full backup:** before apply, recursively copy every file the staging tree will touch into `backup/`. Use staging tree as the manifest of files to back up.
2. **Move-then-copy:** for each file in the staging tree, rename existing destination file to `dst.old`. Copy new. On any failure, rename all `*.old` back. On final success, delete all `*.old`.

**Verify:** Add a file to the staging tree that doesn't currently exist in the install dir (`extras/new_file.txt`). Run apply. Force a mid-apply failure. Assert rollback removes `extras/new_file.txt`.

### 7A.3 [CRITICAL] `verify_installation` must hash files against manifest

**File:** `native/nightshade_native/updater/src/main.rs:273-291`.

**Problem:** Only checks two filenames exist. No SHA-256 against manifest hashes. Boot-time `verifyPendingInstall` only checks version string.

**Fix:**
1. Pass `expected_hashes.json` as an updater argument (Dart side writes it from the verified manifest).
2. After apply, re-hash each file listed; compare to expected. On mismatch: rollback, return error.
3. On boot-time `verifyPendingInstall`, re-hash the executable path against the recorded post-install hash. If mismatch: surface an error and refuse to mark the update verified (force user to re-update or rollback).

**Verify:** Test with a corrupted-but-correct-named file in the staging tree. Assert hash mismatch, assert rollback.

### 7A.4 [CRITICAL] `cleanup_staging` must not delete its parent

**File:** `native/nightshade_native/updater/src/main.rs:386-394`.

**Problem:** `staging_dir.parent()` is `…/updates/`. `fs::remove_dir_all(parent)` wipes `backup/`, `pending_install.json`, and any future-staged update.

**Fix:** Delete `staging_dir` itself (not its parent). The backup directory is removed by next-launch verification logic (see §7A.3) once it confirms the update is healthy.

### 7A.5 [HIGH] `Process.start` → `exit(0)` race

**File:** `packages/nightshade_updater/lib/src/services/update_service.dart:385-389`.

**Problem:** App calls `Process.start` then immediately `exit(0)`. No verification the updater spawned, no log flush. If updater crashes during `OpenProcess`, app exits and user sees nothing.

**Fix:**
1. Verify `pid != 0` from `Process.start`.
2. Sleep 500 ms while logging "updater launched, pid=X".
3. Flush the logger explicitly.
4. On `Process.start` exception, restore status to `staged` and surface the error.

### 7A.6 [HIGH] Add a file lock / mutex to the updater

**File:** `native/nightshade_native/updater/src/main.rs` (entry point).

**Problem:** `pending_install.json` is the only marker; not a lock. Two concurrent updater instances (user clicks Restart twice; or stale-from-prior-crash) race.

**Fix:**
1. On startup, `try_lock` a file at `<install_dir>/updates/.updater.lock` (use `fs2::FileExt::try_lock_exclusive`).
2. If lock fails, log "another updater is running" and exit non-zero.
3. Hold the lock until apply + verify completes; release on exit.

### 7A.7 [HIGH] LAN push: hard cap on incoming package size

**File:** `packages/nightshade_updater/lib/src/services/lan_push_receiver.dart:289-452`.

**Problem:** No size cap independent of the manifest. If a malicious unsigned manifest reaches the size loop (post-signature-verify failure), filesystem can be filled.

**Fix:**
1. Hard cap: 1 GiB (configurable).
2. Verify manifest signature **before** opening `update.zip` for write (already done at line 328-332 — confirm path; ensure no early-write happens before the verify).
3. Add a config check: if no `NIGHTSHADE_UPDATE_PUBLIC_KEY` was compiled in, refuse to start the LAN push receiver and log a clear message ("LAN push receiver disabled: no trusted public key compiled in. Build with --dart-define=NIGHTSHADE_UPDATE_PUBLIC_KEY=... to enable.").

# 8. Strategic feature gaps vs competitors

This section is informational, not a 2.5.0 fix list. It maps the audit-team's gap analysis onto what to build next to widen Nightshade's differentiation. Group these into a v2.6 / v3.0 roadmap discussion, not the v2.5.0 release.

## 8.1 Where Nightshade is already ahead

- **Cross-platform driver breadth.** ASCOM + INDI + Alpaca + 12 native vendor SDKs is unique among shipped products.
- **Headless + web + mobile + WebRTC remote story.** No Windows competitor has the same breadth.
- **Science depth.** Differential photometry, AAVSO/MPC export, transient detection, period analysis, line-ratio products, transparency tracking, PSF field map, moving-object detection. Beyond NINA/SGP/Sharpcap; competitive with AstroImageJ.
- **Trigger system.** HumidityThreshold, FocusDrift (rolling), DomeShutterNotOpen, MountTrackingLost, GuideStarLost as first-class triggers — broader than SGP, on par with NINA.
- **Weather subsystem.** Cloud motion analysis + radar + alerts is meaningfully ahead.
- **Live annotation pipeline.** Real-time star/DSO labeling on live image is rare and valuable.

## 8.2 The 10 highest-impact gaps

Ranked by user-visible impact, then by effort.

| Rank | Gap | Effort | Why it matters | Competitor |
|---|---|---|---|---|
| 1 | **Plugin system is bare-bones.** Logger + storage + eventbus only. No custom sequencer nodes, no UI panels, no manifest with permissions, no marketplace. | L (months) | NINA's plugin economy is its moat. Without a real plugin API, you can never out-feature NINA — only out-polish it. | NINA |
| 2 | **No advanced unattended scheduler / RoboTarget equivalent.** `SchedulerService` does altitude math but no dynamic target rotation engine. | M-L (weeks) | Voyager RoboTarget is $1k+ because there's no free RoboTarget. NINA's Target Scheduler plugin is the most-installed plugin. Owns the unattended-observatory persona. | Voyager, NINA Target Scheduler |
| 3 | **No embedded plate solver.** ASTAP / astrometry.net must be installed separately; no nova.astrometry.net cloud fallback. | M (weeks) | First-light UX disaster: new user gets "ASTAP not found." Sharpcap and Ekos both ship internal solvers (Tetra3, StellarSolver). | Sharpcap, Ekos, ASIAir |
| 4 | **No conditional/loop sequencer nodes for altitude / sun-elevation / moon-distance.** Altitude is a trigger (after the fact), not a precondition. | S-M (days) | NINA's "Loop while sun below -12°" / "Wait until target altitude > 35°" are the building blocks of unattended sequences. | NINA, Voyager |

| 10 | **No defect map / bad-pixel map generation.** Calibration covers darks/flats/bias only. | S (days) | Stacked masters keep visible hot/dead pixel trails. PixInsight CosmeticCorrection handles this; SGP and NINA both build defect maps from dark frames. | NINA, SGP, PixInsight |

## 8.3 Five strategic recommendations

### 8.3.1 Make the plugin system real, then ship 5 reference plugins
Bare APIs don't grow ecosystems. Move beyond log+storage+eventbus to:
- `SequencerNodePlugin`, `PanelPlugin`, `DeviceDriverPlugin`, `ProcessingPlugin` interfaces.
- Manifest with permissions (filesystem, network, devices), version compatibility, signature.
- Plugin store endpoint (a static JSON on a CDN is enough at first).
- Sandbox via Dart isolates or process isolation.
- Build 3–5 first-party "plugins" (transient detection, AAVSO export, weather radar) using the same API to dogfood and prove the API.

This is the single highest-leverage long-term moat.

### 8.3.2 Lean into cross-platform / mobile / headless as the brand
You shipped what NINA hasn't and SGP/Voyager won't ever ship. Stop comparing on Windows-only feature checkboxes. Position as **"the app you can run on your Linux observatory PC, control from your iPhone via WebRTC, and headless-script from a Raspberry Pi."**
- Linux observatory image (deb / rpm / flatpak) and a docker headless image.
- iOS App Store presence, possibly with a Pro subscription funding the project.
- WebRTC + web dashboard hardening (per §2 in this document) until a session can be fully run without the desktop UI open.

### 8.3.3 Build a real RoboTarget-class scheduler, market it as the centerpiece
If Nightshade ships a built-in dynamic scheduler picking targets by altitude + moon + transit + priority + filter-completion-percentage, you instantly own the unattended-observatory persona that Voyager charges $1k for.
- Marry `project_tracking_service.dart` (already present) to `SchedulerService` and a new `DynamicTargetExecutor` in the sequencer.
- Concept: "Project = target + filter quotas. Scheduler picks projects to fill the night."
- v1 with altitude + moon distance is shippable in weeks.

### 8.3.4 Double down on science as the unique brand
You already have a feature set no competitor has. Push it.
- Ship pre-loaded AAVSO target lists, comet ephemerides (auto-updated from MPC), asteroid pre-discovery lists.
- **Automatic supernova/transient hunt mode**: image known galaxies, diff against reference, alert.
- Co-author papers with data scientists. Get cited. Become the academic-hobbyist tool. NINA can't catch up because it's volunteer-built around imaging, not science.

### 8.3.5 Fix the new-user cliff before adding more features
Nightshade has more features than NINA, but a new user's first hour is currently worse:
- Bundle a tiny embedded solver (Tetra3 port or a 100 MB Gaia-DR3 subset) so plate solve "just works." (See §6.1.)
- 7-step interactive "first night" tutorial: connect → polar align → focus → frame → guide → sequence → review.
- 5 sample sequences shipped: DSLR M31, mono LRGB M51, narrowband NGC 7000, lunar imaging, planetary. Discoverable from a "Library" tab.
- Every error message links to a help URL.

# Verification approach

Most of these fixes are verifiable without hardware:

1. **Unit tests** for: HFR median, FWHM formula, VNG threshold, FITS round-trip, WCS parse error propagation, `parse_native` over all device-ID formats, retry-on-trigger paths, checkpoint round-trip.
2. **Simulator backend** at `packages/nightshade_core/test/services/centering_service_test.dart` (existing pattern) covers: meridian flip, centering abort, sequencer recovery actions, flat-wizard event subscription.
3. **Mock device wrappers** for: silent-fallback removal (mount_get_status returning Option), per-device-type discovery error propagation, INDI mount Result-typed methods.
4. **Curl smoke tests** against headless API for: 400 on missing fields, non-200 on failures, CORS rejection of non-allow-listed origins, timing-attack-resistant token comparison microbenchmark.
5. **Browser automation** (or manual) for: dashboard pairing flow, d-pad press-and-hold, mobile QR rejection of malformed payloads.
6. **Manual hardware testing** is irreducible for: real device cooler-on accuracy on ZWO/QHY/Player One; SkyWatcher meridian flip; Alpaca ImageBytes binary protocol against a real Alpaca server; OTA updater on a real install with a locked DLL.
