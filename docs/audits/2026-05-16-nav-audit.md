## Dead routes

1. **`/mobile-dashboard`** at `packages/nightshade_app/lib/router/app_router.dart:55-59`
   - No call site. Grep for `/mobile-dashboard` returns only the route definition.
   - The mobile entry point (`apps/mobile/lib/main.dart:570-580`) builds a standalone `MaterialApp(home: MobileDashboardScreen())` for phones rather than entering the GoRouter at `/mobile-dashboard`. The injection of `mobileDashboardBuilder` at `apps/mobile/lib/main.dart:32` is therefore also unused at runtime.
   - Likely: leftover from an earlier mobile-routing design.

2. **`/diagnostics/dump`** at `packages/nightshade_app/lib/router/app_router.dart:263-271`
   - No call site. Grep for `/diagnostics/dump` or `goNamed('diagnostics-dump')` returns only the route definition.
   - Doc-comment says "support links can deep-link past the settings tree", but nothing in the UI exposes a button. `DiagnosticDumpScreen` is implemented and functional — just unreachable.

3. **`/scheduler`** (legacy redirect) at `packages/nightshade_app/lib/router/app_router.dart:237-246`
   - No `context.go('/scheduler')` / `push('/scheduler')` / `goNamed('scheduler')` call sites remain in `packages/` or `apps/`. The route exists only as a deep-link compatibility shim (comment in router says "kept for one release").
   - The `pageBuilder` constructs a `SchedulerScreen` that is never reached because `redirect` fires first.
   - Note: `/diagnostics` (the other deprecated redirect at line 250-258) has the same shape but is more defensible because users may still have old bookmarks / share-links. `/scheduler` was a side-nav slot that was removed in W8-SCHED-MERGE, so external deep links are less plausible.

## Dead buttons / broken links

1. **`packages/nightshade_app/lib/screens/planner/planner_screen.dart:2089`** — `NightshadeButton(label: 'Open catalog settings', onPressed: () => context.go('/settings/catalogs'))`
   - `/settings/catalogs` is **not a defined route**. The user lands on `/settings` (no error — go_router silently routes to the closest match: the parent `/settings`), but no tab is auto-selected and the catalog UI is several clicks away.
   - Either define the sub-route or change the link to `/settings` plus a tab-query param (mirror the existing `/analytics?tab=…` pattern).

## Missing back / exit affordance

These three routes live inside `AppShell` (so the side nav and bottom nav are still present, giving the user *a* way out), but none of them offer a "return to where you came from" affordance:

1. **`/transients`** — `packages/nightshade_app/lib/screens/transients/transients_screen.dart`
   - No `AppBar`, no back button, no `context.pop()` call anywhere in the file.
   - Reached via `TransientAlertBadge` in the title bar (`title_bar.dart:75`) using `context.go` — so there's literally nothing on the navigator stack to pop. User can only escape by clicking a side-nav entry.
   - Compare with `/polar-alignment`, which has a custom back affordance at `polar_alignment_screen.dart:196-208` that falls back to `/imaging` when `canPop()` is false. `/transients` should do the same — pop back to wherever the badge was clicked, or fall back to `/dashboard`.

2. **`/diagnostics/dump`** — `packages/nightshade_app/lib/screens/diagnostics/diagnostic_dump_screen.dart`
   - No back button, no AppBar, no `pop()`, no `context.go(...)` exit. The screen body is just the bug-bundle UI. Currently moot because nothing links to it (see Dead Route #2), but if the link is wired up it will trap the user.

3. **`/settings/plate-solving`** — `packages/nightshade_app/lib/screens/settings/plate_solving_settings_screen.dart`
   - Does have `AppBar(title: 'Plate Solving')` (line 116-125), and `automaticallyImplyLeading` defaults to true, so an AppBar back button is rendered.
   - BUT: callers use `context.go('/settings/plate-solving')` (e.g. `plate_solver_required_banner.dart:33`), not `context.push`. With `go_router`, hitting the implicit AppBar back button on a route entered via `go` will call `Navigator.pop()` on an empty stack and throw a "no routes on stack" assertion in debug mode, or silently no-op in release. Either:
     - Change the callers to `context.push('/settings/plate-solving')`, or
     - Replace the implicit leading with an explicit one that calls `context.go('/settings')`.

## Orphan screens (not navigation defects, but noted)

It is not wired into any route or widget. Likely a leftover from the W8-SCHED-MERGE / Plan-Tonight consolidation where suggestions moved into a planner tab (the widgets in `suggestions/widgets/` are still used, but the screen wrapper is dead). The file is large (~525 lines) so worth confirming before deletion.

## Notes on routes that look orphan-ish but aren't

- **`/imaging/preview/:imageId`** — only call site is `apps/mobile/lib/services/notification_service.dart:226` (notification deep-link). Reachable; keep.
- **`/tutorial/first-night`** — called from `settings/widgets/help_tutorials_settings.dart:55`. Reachable; keep.
- **`/diagnostics`** (legacy redirect) — same shape as `/scheduler` but more defensible (see Dead Route #3 discussion).
- **`/onboarding`** — auto-launched by `EquipmentOnboardingLauncher` (`widgets/equipment_onboarding_launcher.dart:72`) when no profiles exist + user is on `/dashboard`. Has an explicit `Skip onboarding` exit (header `onExit` callback at `onboarding_screen.dart:84-91`) that goes to `/dashboard`, and finishing the wizard also goes to `/dashboard` (line 99). Verified reachable and exitable.

## Recommendations (concrete)

1. **REC-1** — Delete `/mobile-dashboard` route (lines 31-59 of `app_router.dart`) and the `mobileDashboardBuilder` injection in `apps/mobile/lib/main.dart:30-32`. The phone path uses a standalone MaterialApp; the route is dead weight.

2. **REC-2** — Delete `/scheduler` redirect route (lines 237-246 of `app_router.dart`). No call sites remain in the codebase; only deep-link compatibility for external bookmarks would justify it, and the deprecation window has likely elapsed.

3. **REC-3** — Either wire `/diagnostics/dump` from Settings → Help (next to "First Night Walkthrough") **or** delete the route and `DiagnosticDumpScreen`. Decision should be product-side; do not leave it implemented-but-unreachable.

4. **REC-4** — Fix `planner_screen.dart:2089` to route to a valid catalog destination. Either add a `/settings/catalogs` sub-route (mirroring `/settings/plate-solving`) or change to `context.go('/settings?tab=catalogs')` if catalog mgmt lives as a Settings tab.

5. **REC-5** — Add a back/close affordance to `TransientsScreen`. Pattern from `polar_alignment_screen.dart:196-208` is the right template (`canPop()` → `context.pop()` else `context.go('/dashboard')`). Same for `DiagnosticDumpScreen` if REC-3 keeps it.

6. **REC-6** — Change all callers of `/settings/plate-solving` from `context.go` to `context.push` so the AppBar's auto-back actually has a route to pop. Affected: `plate_solver_required_banner.dart:33` (and any future callers). Alternatively, override `AppBar.leading` to call `context.go('/settings')` explicitly.

7. **REC-7** — Delete `SuggestionsScreen` (orphan; superseded by planner tabs) or wire it into a route. Recommend delete since `widgets/suggestions/` is still imported into the planner.
